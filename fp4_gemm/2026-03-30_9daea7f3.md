两个文件的 scale 计算部分完全一样，差异在于 **FP32 → FP4 值的舍入方式**。

## 核心区别：舍入策略

### `right_fp4.py` — Round-to-Nearest-Even（Banker's Rounding）✅

Normal 路径（第 106-112 行）：
```python
mant_odd = (normal_x >> (MBITS_F32 - MBITS_FP4)) & 1
val_to_add = ((EXP_BIAS_FP4 - EXP_BIAS_FP32) << MBITS_F32) + (1 << 21) - 1
normal_x += val_to_add
normal_x += mant_odd      # 关键：只有结果为奇数时才多加 1
```

逻辑是：
- 先加 `0x1FFFFF`（刚好比一半少 1）
- 再加 `mant_odd`（目标尾数最低位是否为 1）
- 效果：**恰好处于中间值时，只在结果为奇数时才向上进位**，保证结果是偶数 → 这就是 banker's rounding

Denormal 路径（第 93-102 行）利用 FP32 加法天然的 RNE 行为（加一个 magic number），也是 round-to-nearest-even。

### `wrong_fp4.py` — Round-Half-Up（始终向上进位）❌

第 90 行：
```python
combined_val = (((e << 2) | (m >> 21)) + 1) >> 1
```

这里无条件 `+1` 再右移，等价于：**中间值（tie）时一律向上进位**。这是 round-half-up，会引入系统性正偏（systematic positive bias）。

## 结论

| | `right_fp4.py` | `wrong_fp4.py` |
|---|---|---|
| **舍入方式** | Round-to-Nearest-Even (RNE) | Round-Half-Up |
| **Tie 处理** | 向偶数方向进位 | 始终向上 |
| **统计偏差** | 无偏 | 有正偏 |
| **是否符合规范** | ✅ 是 | ❌ 否 |

**`right_fp4.py` 是对的。** OCP MX 规范（Microscaling Formats Specification）要求元素量化使用 **round-to-nearest-even**，也就是你说的 banker's rounding。`wrong_fp4.py` 的 round-half-up 在大量量化时会产生累积的正向偏差，影响模型精度。

好，继续概念 4。我们用 `qx = 3.0` 追踪一遍完整的 Normal 转换过程（先忽略 rounding 部分）。

```
3.0 的 FP32 整数表示：
  S=0  E=1000_0000 (128)  M=100_0000_0000_0000_0000_0000
  
  u32 = 0x 4040_0000
```

目标：得到 FP4 的 `101`（E=10, M=1 → 3.0）。

**第一步**：加 `(-126) << 23` 来转换 exponent bias。

`(-126) << 23` 的效果是把 exponent 字段从 128 减到 `128 - 126 = 2`。

加完后整数表示变成：

```
  S=0  E=0000_0010 (2)  M=100_0000_0000_0000_0000_0000
```

**第二步**：`>> 22` 把 23 位尾数压成 1 位。

右移 22 位后，原来的 bit 22（尾数最高位）变成了 bit 0。

```
  移位前: ...  0000_0010  1  00_0000_0000_0000_0000_0000
                 E(2)     M(1 bit we keep)  (22 bits shifted out)
  
  移位后: ...  0000_0000  0000_0000  0000_0  101
                                              ^^^
                                              E=10, M=1 → FP4 的 3.0 ✓
```

得到 `0b101 = 5`，对应 FP4 的 `10.1` = 3.0。
